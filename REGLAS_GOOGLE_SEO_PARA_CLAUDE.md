# Reglas de Google SEO para proyectos web

## Propósito

Este documento define las reglas que Claude debe seguir al diseñar, desarrollar, revisar u optimizar una página web con enfoque en Google SEO.

Estas reglas deben aplicarse a cualquier proyecto, sin importar el giro del negocio, la tecnología utilizada o la ubicación del cliente.

El objetivo no es “rellenar” una página con palabras clave, sino construir un sitio que:

1. Pueda ser rastreado e indexado por Google.
2. Responda correctamente a la intención de búsqueda del usuario.
3. Sea útil, claro, rápido y confiable.
4. Tenga una arquitectura escalable.
5. Permita medir resultados y mejorar continuamente.
6. Genere oportunidades comerciales reales.

---

# 1. Principios generales

Claude debe seguir siempre estos principios:

- Crear contenido principalmente para personas, no para manipular buscadores.
- Evitar textos genéricos, repetitivos o sin valor.
- No prometer posiciones específicas en Google.
- No usar técnicas de SEO engañosas o de alto riesgo.
- Priorizar claridad, utilidad, confianza y experiencia del usuario.
- No crear páginas duplicadas cambiando únicamente una palabra, ciudad o servicio.
- Cada página debe tener una intención de búsqueda definida.
- Cada URL debe tener un propósito específico.
- Antes de escribir contenido, se debe entender el negocio, cliente, servicio, zona e intención de búsqueda.
- Toda recomendación SEO debe tener una razón técnica, comercial o de experiencia de usuario.

---

# 2. Información mínima del proyecto

Antes de diseñar la estrategia SEO, Claude debe identificar:

## Datos del negocio

- Nombre comercial.
- Dominio principal.
- Descripción del negocio.
- Productos o servicios.
- Diferenciadores.
- Público objetivo.
- Ubicación o zonas de atención.
- Modalidad de servicio: local, nacional, internacional o en línea.
- Objetivo principal del sitio.
- Acciones de conversión esperadas.
- Competidores principales.
- Tecnología utilizada.
- Estado actual del sitio.

## Objetivos posibles

El sitio puede buscar:

- Generar solicitudes de contacto.
- Conseguir llamadas.
- Recibir mensajes por WhatsApp.
- Vender productos.
- Reservar citas.
- Generar registros.
- Obtener visitas a una ubicación.
- Construir autoridad.
- Educar al mercado.
- Posicionar una marca.

Claude debe definir cuál es el objetivo principal y cuáles son secundarios.

---

# 3. Proceso SEO obligatorio

Todo proyecto debe trabajar en el siguiente orden:

1. Diagnóstico del negocio.
2. Auditoría del sitio actual.
3. Investigación de palabras clave.
4. Clasificación por intención de búsqueda.
5. Diseño de arquitectura.
6. Asignación de palabras clave por URL.
7. Optimización SEO on-page.
8. Implementación técnica.
9. Creación de contenido.
10. SEO local, cuando aplique.
11. Medición.
12. Mejora continua.

No se debe comenzar escribiendo páginas o artículos sin definir antes su objetivo e intención de búsqueda.

---

# 4. Investigación de palabras clave

## Reglas

Claude debe investigar términos basados en:

- Problemas del cliente.
- Servicios ofrecidos.
- Productos.
- Ubicación.
- Tipo de negocio.
- Preguntas frecuentes.
- Nivel de conocimiento del usuario.
- Etapa del proceso de compra.

## Tipos de palabras clave

Las palabras clave deben clasificarse como:

### Informativas

El usuario busca aprender.

Ejemplos:

- qué es un respaldo empresarial
- cómo proteger la información de una empresa
- por qué falla una computadora

### Comerciales

El usuario está comparando opciones.

Ejemplos:

- mejor servicio de respaldo para empresas
- planes de mantenimiento para oficinas
- servicio de TI para pequeñas empresas

### Transaccionales

El usuario tiene intención de contratar o comprar.

Ejemplos:

- contratar soporte tecnológico
- empresa de mantenimiento de computadoras
- cotizar respaldo empresarial

### Locales

El usuario busca una solución en una ubicación concreta.

Ejemplos:

- soporte tecnológico en Toluca
- mantenimiento de computadoras en Metepec
- respaldo para empresas en Lerma

### De marca

El usuario busca directamente una empresa o producto.

Ejemplos:

- TekShield Solutions
- TekShield Care
- planes TekShield

## Intención de búsqueda

Claude debe identificar qué espera encontrar el usuario:

- Una definición.
- Una guía.
- Una lista.
- Una comparación.
- Un servicio.
- Un producto.
- Una empresa local.
- Un precio.
- Un contacto.
- Una solución inmediata.

La página debe responder esa intención de manera directa.

---

# 5. Mapeo de palabras clave

Cada palabra clave principal debe asignarse a una sola URL principal.

No se deben crear varias páginas compitiendo por la misma búsqueda.

Para cada URL se debe definir:

- Palabra clave principal.
- Palabras clave secundarias.
- Intención de búsqueda.
- Público objetivo.
- Objetivo comercial.
- Tipo de página.
- Llamado a la acción.
- Enlaces internos necesarios.

Ejemplo:

| URL | Palabra principal | Intención | Objetivo |
|---|---|---|---|
| /respaldos-para-empresas/ | respaldos para empresas | Comercial | Generar contacto |
| /que-es-un-respaldo/ | qué es un respaldo | Informativa | Educar |
| /servicios/toluca/ | soporte tecnológico en Toluca | Local | Generar prospectos |

# 6. Arquitectura del sitio

## Principios

La arquitectura debe ser:

- Clara.
- Jerárquica.
- Fácil de navegar.
- Escalable.
- Comprensible para Google.
- Comprensible para el usuario.
- Con URLs descriptivas.

## Estructura recomendada

```text
/
├── servicios/
│   ├── servicio-principal/
│   ├── servicio-secundario/
│   └── servicio-especializado/
├── sectores/
│   ├── sector-uno/
│   └── sector-dos/
├── ubicaciones/
│   ├── ciudad-uno/
│   └── ciudad-dos/
├── recursos/
│   ├── guias/
│   └── preguntas-frecuentes/
├── blog/
├── nosotros/
└── contacto/
```

## Reglas de URLs

Las URLs deben:

- Ser cortas.
- Ser descriptivas.
- Usar minúsculas.
- Separar palabras con guiones.
- Evitar caracteres especiales.
- Evitar parámetros innecesarios.
- No incluir fechas salvo que sean indispensables.
- No cambiar sin una razón justificada.

Ejemplo correcto:

```text
/respaldos-para-empresas/
```

Ejemplo incorrecto:

```text
/pagina?id=487&servicio=2
```

# 7. Reglas para páginas de servicios

Cada página de servicio debe incluir:

1. Una propuesta clara.
2. El problema que resuelve.
3. A quién está dirigida.
4. Beneficios.
5. Cómo funciona.
6. Alcance del servicio.
7. Diferenciadores.
8. Preguntas frecuentes.
9. Pruebas de confianza.
10. Llamado a la acción.

## Estructura sugerida

```text
H1: Servicio principal y ubicación o mercado

Introducción:
- Qué es.
- Para quién es.
- Qué problema resuelve.

H2: Problemas que resuelve

H2: Beneficios

H2: Cómo funciona

H2: Qué incluye

H2: Por qué elegir a la empresa

H2: Preguntas frecuentes

CTA final
```

El texto debe ser específico. No usar frases vacías como:

- Somos los mejores.
- Calidad garantizada.
- Soluciones innovadoras.
- Servicio de clase mundial.

Toda afirmación debe explicarse o respaldarse.

---

# 8. Reglas para páginas locales

Las páginas locales solo deben crearse si la empresa realmente atiende esa zona.

Cada página debe tener contenido útil y diferente.

Debe incluir:

- Nombre de la ciudad o zona.
- Servicios disponibles.
- Tipo de clientes atendidos.
- Problemas comunes de la zona o mercado.
- Forma de atención.
- Cobertura.
- Tiempos o condiciones, cuando se puedan publicar.
- Pruebas de presencia o experiencia.
- Preguntas locales.
- Llamado a la acción.

No crear páginas idénticas cambiando únicamente el nombre de la ciudad.

---

# 9. SEO on-page

Cada página indexable debe tener:

- Un título SEO único.
- Una metadescripción útil.
- Un H1 único.
- Encabezados jerárquicos.
- Contenido original.
- URL descriptiva.
- Enlaces internos.
- Imágenes optimizadas.
- Texto alternativo cuando corresponda.
- Etiqueta canónica.
- Llamado a la acción.
- Datos estructurados cuando sean útiles.

---

# 10. Títulos SEO

## Reglas

El título debe:

- Describir claramente la página.
- Incluir la palabra clave principal de forma natural.
- Diferenciarse de otras páginas.
- Comunicar el valor.
- Evitar repetir palabras.
- Evitar títulos genéricos.
- Incluir la marca cuando sea útil.

Formato sugerido:

```text
Servicio o tema principal | Marca
```

Ejemplo:

```text
Cobertura tecnológica para PyMEs en Toluca | TekShield
```

No usar títulos como:

```text
Inicio
Servicios
Bienvenido
Página oficial
```

---

# 11. Metadescripciones

La metadescripción debe:

- Resumir la página.
- Explicar el beneficio.
- Ser coherente con el contenido.
- Incluir términos relevantes de forma natural.
- Motivar el clic sin exageraciones.
- Evitar duplicarse entre páginas.

Ejemplo:

```text
Protege la continuidad de tu empresa con respaldos verificados, mantenimiento preventivo y acompañamiento tecnológico para PyMEs.
```

No usar listas artificiales de palabras clave.

---

# 12. Encabezados

## Reglas

- Debe existir un H1 principal.
- El H1 debe describir el tema central.
- Los H2 deben dividir temas importantes.
- Los H3 deben organizar subsecciones.
- No usar encabezados únicamente por tamaño visual.
- No repetir la misma frase en todos los encabezados.
- Mantener una jerarquía lógica.

Ejemplo:

```text
H1: Respaldos para empresas

H2: Por qué tu empresa necesita un respaldo

H2: Qué información se puede proteger

H2: Cómo funciona el servicio

H3: Configuración inicial

H3: Verificación periódica

H2: Preguntas frecuentes
```

---

# 13. Contenido

## Reglas generales

El contenido debe:

- Responder la intención de búsqueda.
- Ser original.
- Ser fácil de entender.
- Resolver dudas reales.
- Evitar relleno.
- Evitar repeticiones.
- Incluir ejemplos.
- Explicar conceptos técnicos.
- Mantener coherencia con el negocio.
- Incluir información verificable.
- Actualizarse cuando sea necesario.

## Prohibiciones

No se debe:

- Copiar contenido de competidores.
- Publicar contenido generado sin revisión.
- Inventar datos, casos, clientes, estadísticas o certificaciones.
- Incluir testimonios falsos.
- Crear artículos únicamente para aumentar el número de páginas.
- Repetir una palabra clave de manera forzada.
- Ocultar texto o enlaces.
- Usar contenido automático sin valor.

---

# 14. Enlaces internos

Cada página debe enlazar a otras páginas relacionadas.

Los enlaces internos deben:

- Ayudar al usuario.
- Mostrar relación entre temas.
- Facilitar el rastreo.
- Usar texto descriptivo.
- Evitar expresiones genéricas como “haz clic aquí”.
- No ser excesivos.
- Apuntar a páginas relevantes.

Ejemplo correcto:

```text
Conoce cómo funciona nuestro servicio de respaldos para empresas.
```

Ejemplo poco útil:

```text
Haz clic aquí.
```

---

# 15. Imágenes

Las imágenes deben:

- Tener un propósito.
- Estar comprimidas.
- Usar formatos modernos cuando sea posible.
- Tener dimensiones adecuadas.
- Evitar archivos demasiado pesados.
- Tener nombres descriptivos.
- Incluir texto alternativo cuando transmitan información.
- No incluir texto alternativo innecesario en elementos decorativos.
- Mantener una proporción correcta.
- Evitar cambios de diseño durante la carga.

Ejemplo de nombre:

```text
respaldo-informacion-empresa.webp
```

Ejemplo de alt:

```text
Equipo de oficina realizando respaldo de información empresarial
```

# 17. Indexación

Una página debe ser indexable cuando:

- Tiene contenido útil.
- Tiene un objetivo.
- Responde una búsqueda.
- No es duplicada.
- No es una página técnica.
- No es una versión de prueba.
- No está vacía.
- No contiene información privada.

Se debe evitar indexar:

- Paneles administrativos.
- Resultados internos de búsqueda.
- Filtros sin valor.
- Páginas duplicadas.
- Páginas de prueba.
- Carritos.
- Procesos de pago.
- URLs con parámetros innecesarios.
- Contenido privado.
- Páginas sin contenido.

---

# 18. Robots.txt

El archivo `robots.txt` debe:

- Permitir el rastreo del contenido público.
- Bloquear áreas administrativas cuando sea necesario.
- Incluir la ubicación del sitemap.
- No utilizarse como mecanismo para proteger información privada.
- No bloquear recursos necesarios para renderizar la página.

Ejemplo:

```text
User-agent: *
Allow: /
Disallow: /admin/
Disallow: /api/private/

Sitemap: https://www.ejemplo.com/sitemap.xml
```

---

# 19. Sitemap XML

El sitemap debe:

- Incluir únicamente URLs canónicas e indexables.
- Actualizarse cuando se agreguen o eliminen páginas.
- Evitar URLs bloqueadas o con `noindex`.
- Evitar errores 404.
- Evitar redirecciones.
- Usar el dominio principal.
- Enviarse a Google Search Console.

---

# 20. Etiquetas canónicas

Cada página debe indicar su URL canónica cuando sea necesario.

La canónica debe:

- Apuntar a la versión principal.
- Usar HTTPS.
- Usar el dominio correcto.
- Evitar apuntar a páginas irrelevantes.
- Ser coherente con el sitemap.
- Ser coherente con los enlaces internos.

---

# 21. Redirecciones

Usar redirección 301 cuando:

- Una URL cambie de forma permanente.
- Se elimine una página con un equivalente relevante.
- Se consoliden páginas duplicadas.
- Cambie el dominio.
- Cambie la arquitectura.

No redirigir todas las páginas eliminadas hacia la página principal.

Cada redirección debe apuntar a la alternativa más relevante.

---

# 22. Errores 404

Una página 404 debe:

- Mostrar claramente que el contenido no existe.
- Mantener el diseño del sitio.
- Ofrecer navegación útil.
- Incluir enlaces a páginas importantes.
- Devolver un estado HTTP 404 real.
- No indexarse.

---

# 23. Rendimiento

Claude debe priorizar:

- Carga rápida.
- Imágenes optimizadas.
- JavaScript limitado.
- CSS eficiente.
- Fuentes optimizadas.
- Carga diferida.
- Caché.
- CDN.
- Evitar dependencias innecesarias.
- Evitar scripts de terceros sin justificación.

No sacrificar rendimiento por animaciones o efectos visuales.

---

# 24. Experiencia móvil

El sitio debe:

- Ser responsive.
- Tener texto legible.
- Tener botones fáciles de tocar.
- Evitar desplazamiento horizontal.
- Evitar ventanas invasivas.
- Cargar correctamente en conexiones móviles.
- Mantener visibles las acciones principales.
- Evitar menús difíciles de usar.

---

# 25. Core Web Vitals

Se deben vigilar:

- LCP: carga del contenido principal.
- INP: capacidad de respuesta.
- CLS: estabilidad visual.

Las mejoras deben enfocarse en:

- Reducir imágenes pesadas.
- Reservar espacio para imágenes.
- Evitar JavaScript bloqueante.
- Reducir tareas largas.
- Optimizar fuentes.
- Minimizar cambios inesperados.

---

# 26. Datos estructurados

Los datos estructurados deben usarse solo cuando representen contenido visible y real.

Tipos posibles:

- Organization.
- LocalBusiness.
- Service.
- Product.
- Article.
- BreadcrumbList.
- FAQPage.
- Review, solo con datos reales y permitidos.

No incluir datos falsos, ocultos o inventados.

---

# 27. SEO local

Cuando el negocio atienda ubicaciones físicas o zonas específicas, Claude debe revisar:

- Perfil de Empresa en Google.
- Nombre correcto del negocio.
- Categoría principal.
- Categorías secundarias.
- Dirección o área de servicio.
- Teléfono.
- Horarios.
- Sitio web.
- Servicios.
- Descripción.
- Fotografías.
- Reseñas.
- Respuestas a reseñas.
- Consistencia de datos.

Los datos de nombre, dirección y teléfono deben ser consistentes en todos los medios.

---

# 28. Reseñas

Las reseñas deben:

- Ser reales.
- Obtenerse de clientes reales.
- Responderse profesionalmente.
- No comprarse.
- No generarse automáticamente.
- No incentivarse de forma engañosa.
- No publicarse desde cuentas falsas.

Las respuestas deben ser personalizadas y no revelar información privada.

---

# 29. Autoridad y enlaces externos

Los enlaces deben obtenerse mediante:

- Contenido útil.
- Directorios legítimos.
- Asociaciones empresariales.
- Cámaras de comercio.
- Proveedores.
- Socios.
- Medios locales.
- Casos de éxito.
- Colaboraciones.
- Recursos especializados.

No usar:

- Compra masiva de enlaces.
- Granjas de enlaces.
- Comentarios spam.
- Directorios de baja calidad.
- Redes privadas engañosas.
- Intercambios artificiales.

---

# 30. Blog y contenido informativo

No crear un blog sin estrategia.

Cada artículo debe responder:

- Qué busca el usuario.
- Por qué es relevante para el negocio.
- Qué página comercial apoya.
- Qué acción debe realizar el lector.
- Qué enlaces internos necesita.
- Cómo se medirá.

## Estructura sugerida

```text
Título claro

Respuesta breve inicial

Desarrollo del tema

Ejemplos

Errores comunes

Recomendaciones

Preguntas frecuentes

Conclusión

Llamado a la acción
```

Eventos recomendados:

```tsx
import type { Metadata } from "next";

  },
};
```

## Alta prioridad

- Sitio no indexable.

- Robots bloqueando.
- Páginas sin título.
- Contenido duplicado.
- Problemas móviles graves.
- Errores 5xx.
- Redirecciones rotas.
- Páginas comerciales faltantes.

## Prioridad media

- Metadescripciones.
- Enlaces internos.
- Contenido poco profundo.
- Datos estructurados.
- Imágenes.
- Arquitectura.
- Páginas locales.

## Prioridad baja

- Ajustes menores de texto.
- Cambios estéticos sin impacto.
- Variaciones pequeñas.
- Optimización de páginas sin demanda.

# 38. Reglas de comunicación

Claude debe:

- Explicar los conceptos sin tecnicismos innecesarios.
- Separar hechos, recomendaciones y supuestos.
- No inventar resultados.
- Indicar cuando se requiere acceso a herramientas.
- Presentar acciones concretas.
- Dar ejemplos aplicados al negocio.
- Priorizar las tareas con mayor impacto.
- Evitar recomendaciones genéricas.
- No sugerir cambios sin explicar su propósito.

# 40. Regla final

Claude debe tratar el SEO como un proceso continuo.

No se considera terminado después de publicar una página.

Cada proyecto debe:

1. Publicar.
2. Verificar indexación.
3. Medir impresiones.
4. Medir clics.
5. Medir conversiones.
6. Identificar oportunidades.
7. Actualizar contenido.
8. Mejorar la experiencia.
9. Repetir el proceso.

El objetivo final no es conseguir tráfico sin sentido.

El objetivo es atraer al público correcto, responder sus necesidades y convertir las visitas orgánicas en oportunidades reales para el negocio.
